What is a FAIR Data Point?

A stable URL that returns machine-readable metadata about one or more datasets — the "business card" of your data, written in RDF.

Three roles in one

  • Metadata API — serves RDF at stable IRIs via content negotiation (Turtle, JSON-LD)
  • Catalogue — organises datasets into a hierarchy
  • FAIR gateway — each resource carries provenance, licence, and links to the data

Standards it uses

DCAT 3 Dublin Core FDP-O ontology SHACL profiles schema.org re3data
Spec: specs.fairdatapoint.org
Reference impl: FAIRDataTeam/FAIRDataPoint

FDP Index

A central registry that knows where all the FDPs are — enabling discovery without centralising the data.

How an FDP registers

It sends one HTTP POST to the index:

POST https://index.fairdatapoint.org/ Content-Type: application/json { "clientUrl": "https://fdp.semscape.org/ ga4gh-rare-disease-trajectories/fdp/" }

The index then harvests metadata from that URL periodically.

What the index stores

  • The FDP root IRI
  • Last-seen timestamp
  • Health / reachability state
  • Harvested catalog + dataset titles

Discovery flow

FDP publishes metadata
↓ POST /entries (ping)
FDP Index harvests & stores
↓ client queries index
Client discovers datasets
↓ dereferences FDP IRI
Client fetches data directly

Virtual Platform

Many FDPs, connected through the index, behave as one queryable platform — without moving or copying any data.

Core idea

Data stays at the source institution. The index holds only metadata. A client resolves the metadata IRI and then fetches the actual distribution directly — no central data warehouse.

In practice for rare diseases

  • Hospital A registers its patient cohort FDP
  • Registry B registers its FDP
  • Researcher queries the index for ORPHA:98895
  • Index returns IRIs for matching datasets at A and B
  • Researcher downloads distributions — federated without a central server

GA4GH connection

GA4GH Data Connect, DRS, and Beacon v2 all rely on the same principle: stable IRIs for data, federated discovery, no data centralisation. FDPs are the FAIR metadata layer on top.

Reference Implementation

The canonical FDP is a Java Spring Boot service maintained by the FAIRDataTeam at github.com/FAIRDataTeam/FAIRDataPoint.

Architecture

  • Java 17 + Spring Boot
  • MongoDB (metadata store)
  • Blazegraph (triple store)
  • Docker Compose deployment
  • Swagger UI for the REST API

Full dynamic REST API: GET /catalog, PUT /dataset/{id}, etc.

What it provides

  • Web UI for data stewards to create/edit metadata
  • Content negotiation (Turtle, JSON-LD, N-Triples)
  • SHACL validation on write
  • Built-in SPARQL endpoint
  • Auto-ping to FDP Index on startup
  • Access control (open / logged-in / metadata-only)

SHACL profiles

Every resource type (Catalog, Dataset, Distribution) has a SHACL shape that defines required fields. Any metadata that fails validation is rejected at write time.

The profiles are themselves published as RDF and versioned — so older clients can still validate against the version they know.

StaticFDP: a Lightweight Implementation

Same metadata model as the reference implementation — but the "server" is a GitHub repository and GitHub Pages. Zero hosting cost, fully version-controlled, GitOps-native.

The pipeline

ORCID login  +  web form
↓
Cloudflare Worker → GitHub Issue
↓
GitHub Actions → Python → RDF/Turtle
↓
Committed to docs/fdp/

Key insight: GitHub Issues are a structured, audited, ORCID-attributed data store. GitHub Actions is the RDF generator. GitHub Pages is the FDP server. No new infrastructure needed.
